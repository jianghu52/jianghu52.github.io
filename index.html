一秒钟的胜利

前几天罗森刚开始有了招商银行的优惠活动。本着又便宜不占王八蛋的原则。体验了一下招商银行的app的扫码支付。之后有了一点感想。

招商银行可以说是我用过的app中做的最好的银行类app了。个人4大行中除了农行的app没下过之外，工行，建行，中国银行的app都有。此外，华夏的，光大的，兴业的，交行的，我都用过。全部银行加起来可能存款都不过万，可是这些年换了公司啦，办个特殊业务啦，总是会碰见这些银行。所以这些银行的app我基本都体验过一次。这里面最好的还是招商银行。

我用招商银行已经超过10年。最初的卡已经丢到不知道什么地方去了。app也是从有了智能机之后一直装着。时不时还升一次级。招商的app可以说现在已经做的很好了。打开速度很快，而且支持指纹开锁，并且切换流畅。但是就是这次的经历，让我心生感慨。

其实说起来事情非常之细小。我在罗森一直是扫支付宝的。支付宝从冷启动大概需要2秒钟，打开支付码基本上不需要时间。然后出示给收银员，然后扫码。招商银行冷启动也就是2秒多一点，但是这个感知并不明显。但是当切换到支付码的时候，还有一秒不到两秒的延迟。

就是这一秒不到两秒的延迟，让我的支付体验非常之差。其实要是没有对比的话，感觉不出来。但是当你用惯了支付宝，微信，那种不管什么时候，切换支付码都是流畅丝滑的情况下，其他的app如果达不到那种流畅度，立马体验就差很多。为什么这一秒这么关键，其实很简单，当你启动app的时候，你心里是有预期的，你知道一个app不可能瞬间被调用起来，所以启动耗时是可以被接受的。但是当你切换支付码的时候，其实你已经默认了这个app上所有的功能都应该已经是完备了。可以随时使用了，这个时候你切换某一个功能还给你卡顿的情况下，你马上就开始由了卡顿的感觉，哪怕这个时间只有短短的1s。

为什么切换支付码尤其要流畅，一方面是因为你后面还有排队的人，前面还有端着扫码枪的收银员，他们都在给你压力，让你尽快完成支付。另外一方面，当支付码出现的时候，就表示在你这方面的所有支付工作已经完成了，你的下一个动作已经可以是展现给收银员了。在此之前，如果整个交易失败，那么原因大多数都在你这边，要么是网络不行啦，要么手机卡啦，或者不能开锁啦。但是当你的手机展现出支付码的那一刻起，你的工作已经完成了。所以从app打开，到支付码出现，这一段时间里面，一定要流畅。

那么这一秒是否容易提升呢？不容易的。看看国内那么多的能有支付功能的app，能做到流畅切换的我现在看到的也就只有支付宝跟微信。为什么。这一秒的提升其实不仅仅代表着app的流畅度，更直接关系到app的投入。要知道，从app冷启动，到切换到支付码，总共给你的时间也就是2s多一点。在这么短的时间内，你要完成检测网络，计算好二维码内容，并且生成图片，这背后需要很强的技术实力，并且需要很专业的测试。这些都意味着大投入。国内能财大气粗的做到这一点的公司。并不多。

短短的一秒，背后就可能就是上百万的投入的差别，带来的就是，我这样的十年招商app的老用户，依然决定除非有便宜可占，否则依然还是微信支付宝为主。连我都这样了，这些银行app的日活，可想而知。

神户钢铁的堕落与日本制造的悲哀

神户钢铁的丑闻到现在已经发酵很久了。本来按照蹭热点的习惯，我早就应该开始写这篇文章。但是事实上我总觉得思考的不那么透彻的情况下，贸贸然的去评论，太过于肤浅。

我知道神户钢铁的时间很长。我的第一份工作就是在钢铁厂里面。当时厂里面的老师傅就说起来说神户钢铁出产的质量是多么多么好，用料实在，而且还把神户钢铁的产品同国内的宝钢，鞍钢，武汉钢铁厂的产品拿到一起评论。那个时候，因为机器本身是日本产的，需要很多的配件，一部分是日本货，一部分是中国货，当时就对日本货有一个直观的感觉，贵，干净，质量好。

后来又有机会在日本工作一年。当时是给一家日本很有名的保险公司干活。这个时候我开始慢慢静下心来体会为什么日本的质量是那么高了。

说个小事。来感受一下日本的质量是如何提升的。

  • 关于广告牌的放置。我所在的公司很大，每天的人流量很多，公司负一层有一个大食堂，很多公司员工都是中午在那里吃饭，但是日本的生意人也看到这个商机，就很喜欢把摊子摆在食堂的出口。日本的摊位通常都会有放置广告牌的习惯，就是那种竖着的牌子。我刚开始去的时候，发现有一家的广告牌总是比别人要靠后一点，很奇怪，后来再仔细观察发现，原来他那个地方是一个拐弯的地方，而这家公司有很多的行动障碍者，需要坐轮椅行动。所以那个地方的广告牌不能太靠前，不然会妨碍到这些行动障碍者。最让我觉得惊讶的是，到了下周（他们是一周来一次）的时候，那个广告牌内容没变，但是字体变大了一些，这样一来，即使是在靠后一点，从视觉角度来说，跟其他的广告牌的字体看起来竟然差不多。

这就是日本人的做事儿方式，一点一点的扣细节，尽善尽美。好不好，当然好。费不费钱，当然费钱。其实日本费钱的地方也很有。比如在中国大部分的楼宇电梯都是声控或者是红外线的，基本上没人的话就不亮的。但是东京的楼梯，（我后来去京都，大阪也一样）所有的公寓的电灯都是彻夜常亮的。但是不可否认的是，在那种灯火通明的情况下，深夜回家总是有一种安全感的。

其实今天神户钢铁遇到的问题，从更本质的角度来说，其实是关于日本制造的附加值的问题。如果有足够充足的利润，我相信神户钢铁不会造假，之所以要造假，说到底就是利润不足。像神户钢铁这样的大型企业，本身不但负有创造利润的压力，更多的是解决就业的压力，换言之，他不能在经济不景气的情况下裁员，或者很难裁员。可是不裁员，这些人的工资怎么办，利润不够了。所以才会有造假问题。

为什么说是日本制造业的悲哀，究其根本，就是因为制造者对于品质的影响在降低，而科技的影响因素在增加。举个简单的例子，
当年我在钢铁厂里面，所有的关键部位的阀门都必须是日本产的，因为日本产的法兰接口比中国产的光滑好几倍。之所以中国产的阀门做不到那么光滑，是因为打磨这种接口需要非常高的经验的人才可以做到，中国的厂子哪怕舍得钱，花得起时间，也没有太多人能做这件事儿。但是现在呢，打磨这个已经全部是机器自动化了，就起平滑度来说，依然没有日本的法兰高，但是已经可以替换很大一部分阀门了。如果你是日本工厂，你怎么提升？你没有做错任何事情，你的人依然技术很好，工作很勤奋。但是就是没办法卖出更多的产品。

我记得不久前任正非先生还说起过日本的产品，他说日本有一个工厂就生产螺母，但是他生产的螺母连航天飞机都在用。我不知道是哪家公司，但是我相信这家公司也一样有跟神户钢铁一样的痛苦，在高端应用上可能还没有敌手，但是在真正的量大的产品上，竞争对手已经产生了。这样一来，除非利润的大头在那些高端的产品上，否则，利润会越来越少，直至入不敷出为止。

但是这个事情讨厌的事情在于，通常情况下，依靠高端产品获得的超额利润总是不会太多的，因为如果太多的情况下，必然会有竞争者看到这部分市场而产生竞争，导致利润下降。只有那些利润不足以吸引更多的竞争者入场，而本身依靠巨大数量在获得足够利润的企业才是这个世界上的大多数。而偏偏，这样的大多数正好是技术的牺牲品。

日本制造业的通常都是所谓的高端，这些高端在以前都是被制造业奉为典范的，产业升级，品质提升，说的就是这些企业。但是看看这些企业现今的状态。当然，这些状态跟他们少裁员其实关系很大，如果他们足够瘦身的话，也许不少公司能活的很好。但是不要忘记了，如果你的企业一点波动都不愿意承担，有活就找人，没活就裁人，就好像富士康那样的。你永远不要指望能做出日本企业那样的让人惊艳的产品。要知道，那些惊艳的产品的产生，通常都是在那些有梦想，并且相信公司能跟他一起实现梦想的职员创造出来的。富士康能有这样的职员么。我很怀疑。

行转列的循环嵌套的实现方式，总结起来，有以下几个固定的套路。

  2. 第二层循环一定是包含【行转列】里面的行的循环。这个循环里面，因为行数不是固定的，所以定义的变量通常是要转换的值。通常只有一个。如TempValue

循环嵌套本身不是用来干行转列的，但是在复杂的行转列的需求下，比如三行三列的定义数据，要转换成九列的情况下，个人感觉没什么SQL文可以实现。不如自己写存储过程来的直观。当然，这是因为的SQL水平只在增删改查的水平，实在无法理解太过深奥的SQL。

java版本的一个坑

记录一个java版本的坑

我一直以为java8小版本号之间是没有什么差异的，结果今天踩了一个巨大的坑才发现不是。
这次的任务比较奇葩，java不能调试，只有实际的运行环境的文件。我们要改的文件最后会打成一个jar包，以jar包的形式被实际环境调用。所以我们只能是通过输出log的方式来实际查看代码运行效果。

在测试环节，我本地的java版本是[1.8.0-14].我修改完代码之后，连整个工程都上传了一次。测试机装的java版本是1.8.0_10。在这个时候测试机跑代码没有问题。但是当我再次上传代码，测试机用eclipse通过svn同步之后，代码是有的，但是打包的时候，就是看不到新的代码。当时幸好改动的地方不多，一点点的排查，最后查到class文件发现了不同。

具体的原因未知，当时为了快点测试，直接就把测试机的java版本直接升到了小版本的14.之后编译一切正常。到现在也不知道为什么会这样，实际看了下源代码，调用的方法属于不建议用的方法，不知道是否跟这个有关，等后期有时间再装一个jdk。看能不能重现这个问题。今天先记录下来，如果下次再遇到这种本机好用，测试机原来好用，新编译文件怎么都不好用的时候，就要考虑下jdk小版本不同的可能性了。

同步hexo遇到的坑

虽然现在看起来，同步的步骤很简单，但是之前碰壁了N次，遇到了好几个坑。记下来，以便以后能跳过去。

1. npm换源

前一台电脑的宽带是联通的，npm虽然慢，好歹还能用。这次的环境宽带是移动的，虽然号称100M。但是用起来完全没有联通的50M的流畅，时不时就断流，要不是我这个小区只有这一个选择，打死我都不用移动的宽带。因为npm没速度，所以准备换淘宝的源。
随便从网上找了一个命令

$ npm install -g cnpm –registry=https://registry.npm.taobao.org

当时也没看，觉得挺对，就用上了。结果一直连不上，我还以为是淘宝的源也不对。后来仔细一看，原来是用cnpm替代了npm。所以如果用上面这条命令，如果在遇到教程，就应该直接自动把npm替换成cnpm。

或者直接替换npm的源。

npm config set registry https://registry.npm.taobao.org

用这个命令之后，后面可以直接使用npm命令。

2. lf will be replaced by crlf

编译文章的时候，报以上的错误。这个错误在我初始的电脑上安装的时候没有报错，当时那台电脑也是win7.至于为什么没报错，我也不知情。但是这个错误其实是跟git有关。而且只出现咋windows系统上。修改方式也简单。

$ rm -rf .git
$ git config –global core.autocrlf false

注意，一定要先删除.git文件之后，再配置git config。不然不好使。

3.hexo deploy not found git

我个人是执行了

npm install hexo-deployer-git –save

就ok了，但是看网上的意思，这个跟没有配置git的环境也有关系。但是因为我配置了，所以这一步坑不是很大。顺手写上git的配置

$ git config –global user.name “runoob”

$ git config –global user.email test@runoob.com

生成ssh

ssh-keygen -t rsa -C “test@runoob.com”

默认保存路径

C:\Users\Administrator.ssh

4.could not read Username for ‘https://github.com‘

这是我最奇怪的一个错误
我原来的配置信息

deploy:   
    type: git   
    repo: https://github.com/jianghu52/jianghu52.github.io
    branch: master

我以为这个是对的，但是在我现在的机器上就是不好使。
改成这样

deploy:
    type: git
    repository: https://jianghu52@github.com/jianghu52/jianghu52.github.io
    branch: master

竟然就对了。我很疑惑到底是什么导致这个差异，可惜到现在也没调查出来。

以上就是遇到的坑。
总结一下大概的流程。

  1. git，node.js 是必须的。
  2. copy原来的scaffolds，source，themes，.gitignore，_config.yml，package.json六个文件，前三个是文件夹，后三个是单文件

  3. 在copy后的文件夹里面执行hexo安装程序

    npm install hexo-cli -g

    注意，最好是换源之后再执行，不然过不去的可能性很大。

  4. hexo server测试过后，直接hexo g，再hexo d。如果一切顺利，那么你应该可以在github上看到你新的blog了。

华为之狼性--从清理40+研发人员说起

华为之狼性–从清理40+研发人员说起

其实知道这个消息已经很久了。前一阵子各种各样的人都开始讨论华为的这个政策。我当时有些看法，但是觉得那个时候还没有酝酿足够，于是拖到现在。

以前一直知道华为很牛逼，但是这一阵子大家都开始说华为怎么怎么牛逼的时候，我反倒很少说华为牛逼了。但是这次的事情，总让我觉得有不吐不快的地方。所以正好借着新开始折腾博客的机会，开始吐槽一下。

我也不知道这个博客能活多久，更不知道我能坚持多久。但是我会尽可能的坚持下去。

华为的狼性现在已经不是什么新闻了，事实上到了现在，已经很少再有报道提到“狼性”这个词了。但是就如同老话讲的一样，只有起错的名字，没有起错的绰号。我们给任何企业加的标签，如果能被大众所接受。一定是因为大众真的觉得这个标签符合企业在大众心中的定位。

说起狼。有人知道狼团队协作能力强，有战斗力，但是不知道有多少人知道狼群的淘汰也是非常残忍的。当一个狼群里面的食物不在充足到使所有狼都吃饱的时候，老狼就开始被驱离狼群了。不管这个老狼之前在队伍中是什么样的角色，不管他是否还胜任这个职位，只因为他老了。于是他就要被驱离。老实说。华为干这件事儿早已经不是第一次了。07年的时候，8周年大裁员就有这么干过，只不过那时候华为还不够出名，人还不够多，所以大家没啥感觉而已。事实上到了今天，很多人还是没有什么感觉。

之所以到了今天，我依然对于华为裁员耿耿于怀，只是觉得，原来华为还没有我想的那么强大。前一阵子，任正非刚开始说，华为要开始接受失败，开始向无人区探索。这才过了几个月，就要开始裁研发人员了。一个真正好的企业，其实应该是盼着经济危机的。因为一到经济危机，他的对手就少了。而且可以吸收更多的人才，在核心竞争力方面更加的打好护城河。可是华为呢，也跟其他的人一样，开始裁员度日，当然，主动裁员可能给的补偿不错。但是问题是，他裁掉的不是边缘的人，而是核心的研发人员。他这么做，裁掉的不仅仅是成本，更是信心。如果在华为，奋斗是为了钱，不是理想，或者换个说法，大家在华为奋斗，就是为了在被裁之前达到财务自由，那么请问，华为如果有一天不能这么赚钱了，给不到大家这么高的薪水了，或者说到40岁之前没办法让你财务自由的时候，华为还能剩下什么呢？一旦华为开始露出这样的征兆的时候，30+的人会怎么选择，20+的人在招聘的时候会怎么打算？华为已经是我心目中非常优秀的互联网企业了。如果这么优秀的企业，都无法承载人的理想，那么这真是中国企业的一种悲哀。
